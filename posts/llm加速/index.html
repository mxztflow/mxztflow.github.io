1. Transformer Alternates#

1.1 KV-cache#

1.1.1 generator推理过程#

​ kv-cache存在于推理过程token生成的步骤中,decode在generator过程中以auto-regressive的方式运行,如图是GPT-2文本生成的示例.

​ 这种自回归的方式会在有的input的基础上进行推理,得到生成的next_token,将得到的next_token加入input之后再进行推理生成,因此会产生一些重复的动作,从下面的图可以看出:

​ 由于decode-only模型是一个因果模型(当前的输入只和之前的输入相关)在这个过程中,我们会看到每进行一个step,都需要计算将前面的q,k,v会再计算一遍,而实际上我们仅仅只是需要新产生的query.

​ 因此我们会有一个优化的思路,每次我们并不需要再次计算之前已经算过的kv的值,我们只需要分配一个空间将这个保存一下,下次来我们直接用就可以,因此可以引出kv-cache.

​ 没有KV-cache:那么序列长度为k,维度为d的x的线性映射的时间复杂度为$o(kd^2)$,需要的额外空间复杂度为$o(1)$因此当输入序列的长度越来越长的时候,时间复杂度线性方式的增加,会大大增加推理的时间

​ 有了KV-cache如果我们将以前的KV-cache保存下来的话,那么我们每次只需要将以前的kv对导入,然后拼接最新产生的,那么会将时间复杂度减小到$o(d^2)$,会大大减少运行的时间,但是需要一个$o(kd)$空间来存储kv对.KV-cache这是一种用空间换时间的策略.

1.1.2 kv-cache代码#

class KVCache(nn.Module): ## 可以封装为一个类型,在每一个Attention设定好
    def __init__(self, batch_size , seq_length , n_kv_heads , head_dim , dtype , device):
        super().__init__()
        cache_shape = (batch_size , seq_length , n_kv_heads , head_dim)
        self.register_buffer('cache_k' , torch.zeros(cache_shape , dtype=dtype , device=device))
        self.register_buffer('cache_v' , torch.zeros(cache_shape , dtype=dtype , device=device))
    def update(self , start_pos , xk ,xv):
        seqlen = xk.size(1)
        self.cache_k[: , start_pos : start_pos + seqlen] = xk
        self.cache_v[: , start_pos : start_pos + seqlen] = xv
        xk = self.cache_k[: , : start_pos + seqlen]
        xv = self.cache_c[: , : start_pos + seqlen]
        return xk , xv

1.1.3 LLM训练和图解#

  • 训练时,QKV满维度算Attention输入序列长度的不同长度的所有的next_token
  • 推理时,要预测下一个token只需要最后的一个q
  • 因此我们只需要使用,Q[-1],K[:],V[:],就可以计算next_token

1.1.4 KV-cache流程#

  • Without KV-cache每次需要计算全Wq(x),Wk(x),Wv(x),每次计算需要全部变量Attention
  • WithKV-cache,第一步,计算完整的Attention,将kv保存为KV-cache
  • WithKV-cache,第二步,取第一步的next_token计算qkv,将[KV-cache , KV]联合起来,计算qkv
  • KV-cache每个循环会同上一个进行拼接,memory内存为2 * Layer * dim * seq_len

1.1.5 KV-cache显存占用#

  • KV-cache计算量
    • 2 x Batch_size x Layer x seq_length x [dim x nums_heads] x k-bits
  • Existing System Memory中KV cache >30%占用
  • vLLM在40GB能跑30+个batchsize，800+ token/s输出，相较baseline <10 batchsize, 300+token/s输出
  • 现GPU KV Cache中有效存储率低
### 1.1.6 KV-cache推理优化
  • 2 x Batch_size x Layer x seq_length x [dim x n_kv_heads] x k-bits
  1. n_kv_heads:MQA/GQA 通过减少KV的头数减少显存的占用
  2. Seq_length:通过减少序列的长度L,来减少显存的占用,如windows Attn/sink Attn…
  3. KV-cache:从显存的角度去减少空间的利用率,例如Page-Attention
  4. K-bits:从量化上,减少KV-Cache的宽度,例如LLM-QAT进行量化

1.1.6.1 减少头数 MQA/GQA#

  • 一种容易想到的想法是，我们减少KV-heads的数量，对于MQA来说，KV-heads from 8 -> 1 ，那么显存的数量只占原来$\frac{1}{7}$，但是造成的精度影响会比较大。

  • 对于GQA(Grouped-Query_Attention)来说，使得精度提高了，但是速度下降了，例如将KV-heads from 8 -> N。

1.1.6.2 减少KV-cache的length#

  • 减少KV-cache的长度，base窗口：Mistral-7B、StreamingLLM、LongLoRA

  • Dense Attention：初始的Attention的方法，预测最新的token依赖于之前所有的tokens,的出来的结果，所得结果的时间复杂度很高，并且随着输入越长，那么模型的PPL会越来越大。
  • Window Attention：相较于原始的Attention，这个方法每次预测的时候，只关注前若干个tokens，这样的化输出的结果只需关注之前的，而不需要关注最开始的，这样的时间复杂度会大大降低，但是PPl没有明显的下降。
  • Sliding Window/Re-computation：这个直接放弃了KV-cache那么造成的结果就是，它每次计算的之后只需要关注前面的若干个，并且每次都是从当前token往前数若干个，从那个token开始计算，直到预测出来最新的token，通过这种方法的出来的当前的PPl是比较低的，但是时间复杂度会提高一个数量级。
  • Streaming LLM：这个方法的核心在于，再计算的过程当中，保留最开始的若干个token，这个依据来源于作者对多层Attention进行可视化的观察而启发而来(最开始层的注意力最高值集中在距离当前token最近的地方，随着 层数的增加，注意力的峰值会出现在最开始的token处)，那么也就是说，存在最开始的token的话，会让模型对当前的输出结果越发肯定。
